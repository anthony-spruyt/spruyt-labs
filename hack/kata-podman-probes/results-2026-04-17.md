### Probe 1 — hypervisor swap (CLH -> QEMU) [BLOCKED]

Created `RuntimeClass/kata-qemu` (handler=`kata-qemu`) via Flux. It reconciled
cleanly, but no node advertises a CRI runtime named `kata-qemu` -- the
`siderolabs/kata-containers` Talos system extension currently ships only the
default `kata` (cloud-hypervisor) handler. Pod stayed Pending with repeated
`FailedCreatePodSandBox` events; the userns reproducer never ran.

**Rollback:** `RuntimeClass/kata-qemu` revert committed locally and pushed so
Flux removes the orphan. No runtime handler side-effects remain.

**Next step options (user decision):**

1. Register the `kata-qemu` handler in CRI via Talos patch (add a
   `plugins."io.containerd.grpc.v1.cri".containerd.runtimes.kata-qemu`
   block, or equivalent CRI-O `/etc/cri/conf.d/` patch) and re-run probe 1.
   This likely requires a Talos `extraManifests`/patch change + node reboot.
2. Skip probe 1 (hypervisor swap) and proceed to Task 4 (probe 2).
   Rationale: Kata's CLH and QEMU shims use the same guest kernel image,
   so a userns kernel gate is unlikely to differ between them. The
   information value of probe 1 is low compared with the patch+reboot cost.

**Interpretation:**

- `kernel_params` annotation is on kata's default `enable_annotations`
  whitelist -- no Talos config change needed. Params appear verbatim at the
  end of the guest `/proc/cmdline`.
- `CONFIG_USER_NS=y` confirms user namespaces are compiled in.
- `/proc/sys/kernel/unprivileged_userns_clone` is absent -- upstream kernel
  removed this sysctl years ago; the `sysctl.kernel.unprivileged_userns_clone=1`
  boot param is a no-op on the current Kata guest kernel.
- `max_user_namespaces=7943` -- non-zero, so nesting is globally allowed.
- EPERM persists regardless, so the gate is **not** in the guest kernel
  itself. The block is higher up: either kata-agent's OCI-spec enforcement
  of the container uid/gid mapping, or a missing capability/mount-level
  restriction inside the Kata guest container namespace (e.g. container
  user namespace not in "owner" relationship required by kernel to allow
  nested userns mapping write).

**Next step:** probe 2 (deferred) -- inspect kata-agent / OCI spec handling
of userns inside the guest, and/or try probe 4 (privileged sidecar) to see
if the EPERM survives full capability set, which would isolate the gate
to kernel vs. agent policy.
